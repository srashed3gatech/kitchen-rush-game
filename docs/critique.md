### 1.4 Customer arrival arithmetic — does 1 Hz handle peak?

The user asked specifically for this arithmetic check, so let's do it.

- design.md §1: 1 in-game hour = 12.5 real seconds.
- design.md §5.1 peak: 5 customers/in-game-hour at noon.
- 5 customers ÷ 12.5 real seconds = **0.4 customers/real-second**.
- architect.md §4.1: tick is 1 Hz. Per-tick spawn probability at peak ≈ 0.4.

So on a typical noon real-second the spawn coin lands heads 40% of the time. **The numbers do align** — a Poisson with λ=0.4/tick yields ~5 arrivals/12.5s on average. ✅ for the peak.

But: at the trickle end (design.md §5.1 hour 8 AM = 0.5/hr), per-tick probability is ~0.04. That's fine. At quiet-hours minimum (4 AM = 0.3/hr) the probability is ~0.024. Also fine.

**However:** with `setInterval(tickAll, 1000)` (architect §4.1), spawn granularity is coarse. At lunch peak (0.4/tick) the inter-arrival variance is high — sometimes 0 customers for 4 seconds, sometimes 2 in one second. The designer's §12 Q1 explicitly asks for "per-event scheduling for arrivals + 250ms world tick." The architect ignored that preference and used 1 Hz uniformly.

**Resolution.** Keep the 1 Hz world tick (the architect's reasoning about persistence cadence and snapshot polling is sound), but **switch customer arrivals to per-event scheduling**: at the start of each in-game hour, sample arrivals as a Poisson process and queue them with `setTimeout` to fire at the precomputed real-second offsets. Simple, ~30 LOC, and gives smooth arrivals. This is purely an internal `sim/customers.ts` refactor and changes no contract.

**Owner of the fix:** architecture.md §4.1 step 2 — add a sentence.

### 2.1 Coaching cooldowns — **borderline**

design.md §4.3: cooldowns 8–15 real seconds per phrase per worker. Designer's §11 audit says this prevents spam; that's fine for spam-prevention. But there's a UI risk: if the owner clicks a worker and sees a *greyed-out button with a number*, that's a timer.

### 2.2 Mood decay is potentially compounding — **borderline**

design.md §4.1: Level-1 worker decays -3 mood per in-game hour. 21 open hours × -3 = -63 mood/day for an unbroken Level 1. Starting mood 70 → ends day at 7. Next morning's mood is not reset by design §4 (no spec). If mood carries over and decay continues, the worker hits 0 by lunchtime day 2.

The struggling-worker trigger fires at "mood < 30 at any closing time" — easy to hit. Then quiet-hours conversation gives +20/+10/+25 mood. Maximum recovery = +25 mood/night = recovers only ~8 in-game-hours of decay. A new hire could enter a permanent low-mood spiral that *feels* like a failure cascade even though no fail state exists.

**Fix.** Apply two changes in design.md §4:
1. Set a morning mood floor: on day rollover, mood = max(current, 50). Cozy "morning resets you a bit."
2. Mood does not decay while idle (no customer in queue). Decay only applies during active cook ticks. This naturally caps daily decay at ~20 for a normal day.

### 2.3 Speech bubble 4-second timer — fine but tag it

design.md §4.3: "A speech bubble appears above the worker for 4 real seconds." This is fine — it's the bubble disappearing, not a player-action deadline. No fix needed; just note in critique that implementation must not show a depleting bar.

### 2.4 "Struggling worker" blue heart-cloud icon — **borderline**

design.md §4.4: flagged workers display a "small soft-blue heart-cloud icon." Designer frames it as supportive. But to a new player who doesn't yet know the system, seeing a sad-looking icon over a worker's head reads as *failure*. The owner's first reaction will be "what did I do wrong?" — which is the exact feeling we promised to avoid.

**Fix.** Two options:

- **(B)** Make the icon a positive-shaped sprite — a small thought bubble that says "could use a kind word," matched to the coaching tone.

I prefer (A). Open-hours canvas should celebrate, not diagnose.

## 3. Architectural risks

### 3.1 1 Hz full-state polling — will it scale?

User asks: estimate state size at full restaurant.

Realistic Phase-1 fully-staffed restaurant:
- 1 restaurant row (~12 fields) ≈ 200 B JSON
- 6 workers × ~10 fields ≈ 1.2 KB
- ~30 open orders × ~9 fields ≈ 3 KB (realistic open-order count is more like 5–10 at peak, but worst case 30)
- last 3 review IDs ≈ 60 B
- Total: **~4.5 KB JSON per poll**

At 1 Hz, that's ~4.5 KB/s per client. Single-machine, single-player MVP: no problem. The architect's claim of "~2–5 KB JSON" matches.

**Risk:** the polling endpoint is implemented as four SELECTs (restaurants, workers, orders, reviews). better-sqlite3 handles each in <1 ms, so total <5 ms — well inside the 1 s budget.

**Recommendation.** Add a simple ETag/If-None-Match on `/api/restaurant/state` so polls return 304 when nothing changed (e.g. during quiet hours). Cuts the bytes-per-poll to near zero when idle. ~20 LOC. Add to architecture §4.2.

**Not a blocker** — file as a Phase-2 polish item.

### 3.2 AES-256-GCM with auto-bootstrapped master key — what's the failure mode?

architecture.md §5: on first boot in dev, the server generates `MASTER_ENCRYPTION_KEY` and **writes it into `.env`** in the repo root.

Failure modes:
1. **Dev commits `.env` by accident.** `.gitignore` is supposed to catch it but a forced `git add -f .env` would slip through. **Mitigation:** add a pre-commit-style check: if `.env` exists at repo root *with a populated MASTER_ENCRYPTION_KEY*, the server logs a big warning on boot ("This key encrypts your Claude API key; do not commit `.env`"). Better: write the key to `~/.config/kitchen-rush/master.key` outside the repo. Local-dev only — DECISIONS §9 — so a home-dir file is fine.
2. **Shared dev machine.** Another user on the same Mac can `cat ~/.config/kitchen-rush/master.key` and decrypt the SQLite file. Not really a security boundary we promised — DECISIONS §7 already says "Recognise this is insecure." Just document it.
3. **Key rotation.** If the master key is regenerated (e.g. user deletes `.env`), every owner's stored Claude API key becomes unrecoverable. The reviewScorer must gracefully degrade ("decryption failed → use heuristic"). Currently the architect's code path throws in `masterKey()`.

**Fix.** Two changes to architecture.md §5:
- Write master key to `~/.config/kitchen-rush/master.key` (mode 0600), not repo `.env`.
- In `reviewScorer.ts`, wrap `decrypt(...)` in try/catch — on failure, log warning and fall back to mock.

### 3.3 Heuristic fallback scorer — does "feels alive" emerge from templates?

User specifically asks: how many template variants does the designer need?

Math. design.md §6.3 lists 5 phrase banks of 4–5 items each. The composite review template is `{opener} {experience_clause} {service_clause} {closer}` per design §6.1. With 6 personas × 5 wait phrases × 5 food phrases × 5 vibe phrases × 4 cleanliness phrases × 4 service phrases = **~12,000 combinations**, of which only 5–10% are coherent (most combos will sound mechanical or self-contradictory). Realistically ~600 unique-feeling reviews before repetition becomes obvious.

At ~50 reviews/day, that's **12 days of play** before a returning player starts spotting duplicates. Acceptable for MVP.

**Concrete recommendation.** Designer should write:
- **8 phrase variants per bank** (currently 4–5)
- **6 opener phrases per persona** (currently 1 implicit per template)
- **6 closer phrases per persona** (currently 1)

Total designer cost: ~3 hours of writing. That bumps the unique-feeling pool to ~3,000 reviews, or ~60 in-game days of play. Add as a deliverable to design.md §6.3.

**Independent risk:** the heuristic *scorer* (research §3.7) is keyword-based. If the designer expands phrase banks but the heuristic scorer doesn't learn the new keywords, fallback quality drops. Mitigation: lock the keyword vocabulary in design.md §6.3 (use exact phrases the scorer can grep for).

## 7. Verdict: ready to build?

### **BLOCK — but only barely.** Three must-fix items before Phase 3 SWE work starts.

The Phase 1 docs are good. The research is right, the architecture is solid for an MVP, and the design has been audited against the GDD with admirable rigour. None of the contradictions are existential — but #1 below would cause every implementation SWE to write a different scale into their PR.

### Top 3 must-fix items

1. **Lock the score scale at 0–100 integer.** (See §1.1) Cite: design.md §6.4, §7, §10 must be rewritten to match architecture.md's 0..100. Without this, the scorer, the SQLite CHECK constraints, the leaderboard formula, and the heuristic fallback all disagree. **Owner: designer.**

2. **Fix the coaching schema.** (See §1.5) architecture.md §2 `coaching_sessions` requires `worker_response NOT NULL` and implies a Claude call per coaching click. Designer §4.3 locks preset phrases with no per-click Claude call. Add a `kind` discriminator column, allow `worker_response` to be NULL for preset rows, and remove the per-click Claude path from architecture.md §6. **Owner: architect.**

Replace with a `daily_scores` table populated by a server-side function at day-end. **Owner: architect; secondary: designer confirms §10 formula.**

After those three: the rest of §1 (12 more contradictions) and §5 (9 missing pieces) are real but tractable inside the SWE workstream. Tag them as "must-fix-during-implementation" rather than "must-fix-before-implementation."
