# Kitchen Rush — Locked Decisions

This file is the **source of truth** for every agent working on this project.
Every other doc (`docs/research.md`, `docs/architecture.md`, `docs/design.md`, code, etc.) must respect these decisions. If a decision needs to change, update this file first.

Read those first. The decisions below override them where they conflict.

## 1. Scope
- **MVP only (Phase 1).** Single owner runs a beachfront shop. No real-time multiplayer in MVP.
- Future phases (real-time multiplayer, events, full progression) are explicitly **out of scope** for this build but the architecture should not preclude them.

## 2. Roles
- **Owner** — the only human user role in MVP. Logs in by username.
- **Worker** — **NPC only.** AI-driven characters. The data model may leave room for a future human-Worker role, but no UI for it.
- **Customer** — **NPC only.** Generated by the server. They arrive, order, eat, leave a free-text review which Claude scores into the 6 dimensions.

## 3. Tech stack (locked)
- **Frontend:** React 18 + TypeScript + Vite + Tailwind CSS + HTML5 Canvas (no Phaser, no PixiJS in MVP).
- **Backend:** Node.js + TypeScript + Express + SQLite (`better-sqlite3`).
- **Shared types:** A `packages/shared` workspace package.
- **Workspaces:** npm workspaces (no yarn, no pnpm, no turbo).
- **No Redis, no Postgres** in MVP. Migration path acknowledged.

## 4. Game timing
- **One in-game day = 5 real minutes** (8 AM → 5 AM next day spans 5 real min, then 5 AM → 8 AM quiet hours).
- No countdown timers ever shown to the player. No fail state. Quality > speed.

## 5. Visual style
- **Free CC0 sprite pack** (e.g. Kenney.nl). Research agent picks one pack and justifies.
- Top-down 2D, beachfront setting visible from the restaurant.
- Day-night gradient overlay on the scene.

## 6. Tuning & unspecified game numbers
- **Agents decide the numbers.** Provide sensible playtested-feeling defaults. Document them in `docs/design.md`. No `TODO`s left in shipped code.

## 7. Authentication
- **Username-only, no password.** Register a name → that's your account. Session via a signed cookie or simple bearer token in localStorage — architect picks one.
- Recognise this is insecure; do not pretend otherwise. Add a brief `// SECURITY:` comment where appropriate.

## 8. Claude API key handling
- **Owner pastes their own Anthropic key in a settings UI.**
- Server stores it **encrypted at rest** (AES-GCM with a per-server master key from env). Decrypted only at request-time.
- **No bundled key** in the repo or .env.example.
- If no key is set, the AI scoring service must **gracefully fall back** to a deterministic/heuristic scorer so the rest of the game still works.

## 9. Deployment
- **Local dev only.** One command brings up frontend + backend (e.g. `npm run dev`).
- No Docker, no hosting, no CI in MVP.

## 10. Non-negotiable design guard rails
These come from GDD §2, §4.2, §9.2, §10 and override anything else:
1. **No time pressure.** No countdown timers, no expiring orders, no "rush" mechanics.
2. **No fail state.** Sales can be low, money can be tight; you can never lose the game.
3. **Mentorship over management.** Workers cannot be fired. Underperformance → coaching session.
4. **Quality over speed.** Worker speed improves through coaching over time; it is never forced.
5. **Beachfront, cozy, calm.** Visuals and audio should support this tone.

If any agent designs something that violates these, the critique agent must call it out.

## 11. Model allocation
- **Opus 4.7** for decision/intelligence work: research, architecture, design, debate, integration coordination, **and the in-game coaching one-on-one replies** (rare, creative, tone-sensitive).
- **Sonnet 4.6** for coding & heavy token work: all implementation SWE agents, test writing, bug fixing, mechanical doc patching.
- **Haiku 4.5** for narrow classification tasks at runtime: **review scoring via `tool_use`** (high-volume, low-judgment — ~50 calls per in-game day per restaurant; ~$0.001/review per critique §1.2 and `docs/research.md` §3).
